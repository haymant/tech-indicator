# Requirements — Strategy Management, Signal Generation & Backtesting (F-005)

## 1. Business Context

The tech-indicator API currently computes and stores 87+ technical indicators (F-003) but has **no concept of strategies, signals, or backtesting**. Users must manually interpret indicator values and decide when to enter or exit positions — there is no way to:

- Define a trading strategy (e.g., "Buy when RSI(14) < 30 and SMA(20) > SMA(50)")
- Persist strategy definitions for reuse
- Generate actionable buy/sell/hold signals from a strategy
- Run historical backtesting to evaluate strategy performance
- Store and query performance/risk metrics

The upstream library (`github.com/cinar/indicator/v2`) already provides a rich strategy framework with **30+ pre-built strategies** across trend, momentum, volatility, and volume categories, plus a backtesting engine with `DataReport` output. This feature wraps those capabilities into REST endpoints and MCP tools so users can build, test, and iterate on trading strategies programmatically.

The daily trading workflow this supports:

1. **Define a strategy** — pick a strategy type (e.g., `rsi_strategy`, `golden_cross_strategy`) and configure its parameters.
2. **Generate signals** — run the strategy against synced OHLCV data to produce a time series of Buy/Sell/Hold actions. **If signals already exist for the same strategy + underlying + time range, skip recomputation** (idempotent).
3. **Backtest** — simulate the strategy's historical performance to capture total return, max drawdown, Sharpe ratio, win rate, and other risk metrics.
4. **Query results** — filter signals and backtest results by strategy, underlying asset, date range, etc.

These capabilities bridge the gap between raw indicators and actionable trading decisions, making the API useful for daily trading workflows.

## 2. Constraints

| # | Constraint | Rationale |
|---|-----------|-----------|
| C-01 | **Do not modify the `indicator/` folder.** | Upstream library at `github.com/cinar/indicator/v2`. Same constraint as F-001 through F-004. |
| C-02 | **MotherDuck (PostgreSQL wire protocol) is the data store.** | Existing data warehouse. All new tables go there. |
| C-03 | **New DB tables: `strategies`, `signals`, `backtest_results`.** | Keep strategy/signal/backtest data isolated from indicator data. |
| C-04 | **REST + MCP parity.** | Every new endpoint must have a corresponding MCP tool. Same pattern as F-004. |
| C-05 | **Bearer token auth required for mutation endpoints.** | Same `TECH_INDICATOR_API_KEY` pattern as existing POST endpoints. GET endpoints may be unauthenticated for read-only access. |
| C-06 | **Idempotent signal generation.** | If signals already exist for a given strategy_id + underlying + time range, return existing signals instead of recomputing. |
| C-07 | **No CGo or external native code.** | Pure Go — Vercel compatibility. The cinar/indicator library is pure Go. |
| C-08 | **All existing tests must pass unchanged.** | Backward compatibility guarantee. |
| C-09 | **Asset names stored/queried in uppercase.** | Same case-normalization convention as F-001 through F-004. |

## 3. New Database Tables

### 3.1 `strategies`

Persists user-defined strategy configurations for reuse and tracking.

| Column | Type | Description |
|--------|------|-------------|
| `id` | `SERIAL PRIMARY KEY` | Auto-increment ID |
| `name` | `VARCHAR(255) NOT NULL` | User-given name (e.g., "My RSI Strategy") |
| `strategy_type` | `VARCHAR(100) NOT NULL` | Strategy type key from the library (e.g., `rsi_strategy`, `golden_cross_strategy`, `buy_and_hold_strategy`) |
| `underlying` | `VARCHAR(20) NOT NULL` | Asset symbol, uppercased (e.g., `AAPL`) |
| `timeframe` | `VARCHAR(10) NOT NULL DEFAULT '1d'` | Data interval (e.g., `1d`, `1h`) |
| `lookback_days` | `INTEGER NOT NULL DEFAULT 365` | Number of lookback days to use |
| `parameters` | `JSONB` | Strategy-specific parameters (e.g., `{"period": 14}` for RSI, `{"fast_period": 12, "slow_period": 26}` for MACD) |
| `created_at` | `TIMESTAMP NOT NULL DEFAULT NOW()` | Creation timestamp |
| `updated_at` | `TIMESTAMP NOT NULL DEFAULT NOW()` | Last update timestamp |

**Unique constraint:** `(name)` — strategy names must be unique within the system.

### 3.2 `signals`

Records buy/sell/hold signals generated by running a strategy against market data.

| Column | Type | Description |
|--------|------|-------------|
| `id` | `SERIAL PRIMARY KEY` | Auto-increment ID |
| `strategy_id` | `INTEGER NOT NULL REFERENCES strategies(id)` | FK to the strategy that generated this signal |
| `strategy_type` | `VARCHAR(100) NOT NULL` | Denormalized strategy type for fast filtering |
| `underlying` | `VARCHAR(20) NOT NULL` | Asset symbol, uppercased |
| `signal_date` | `DATE NOT NULL` | Date of the signal |
| `action` | `VARCHAR(10) NOT NULL` | One of: `buy`, `sell`, `hold` |
| `price` | `DOUBLE PRECISION` | Closing price at signal time |
| `created_at` | `TIMESTAMP NOT NULL DEFAULT NOW()` | Record creation timestamp |

**Unique constraint:** `(strategy_id, underlying, signal_date)` — prevents duplicate signals for the same strategy + asset + date. This is the idempotency key: if signals exist for a given strategy_id + underlying + date range, the system skips regeneration.

**Index:** `(strategy_id, underlying, signal_date)` for efficient filtered queries.

### 3.3 `backtest_results`

Captures performance and risk metrics from running a strategy backtest.

| Column | Type | Description |
|--------|------|-------------|
| `id` | `SERIAL PRIMARY KEY` | Auto-increment ID |
| `strategy_id` | `INTEGER NOT NULL REFERENCES strategies(id)` | FK to the strategy that was backtested |
| `strategy_type` | `VARCHAR(100) NOT NULL` | Denormalized strategy type |
| `underlying` | `VARCHAR(20) NOT NULL` | Asset symbol, uppercased |
| `start_date` | `DATE NOT NULL` | Backtest start date |
| `end_date` | `DATE NOT NULL` | Backtest end date |
| `total_return` | `DOUBLE PRECISION` | Total return as a decimal (e.g., 0.15 = 15%) |
| `max_drawdown` | `DOUBLE PRECISION` | Maximum drawdown as a decimal |
| `sharpe_ratio` | `DOUBLE PRECISION` | Sharpe ratio (annualized) |
| `win_rate` | `DOUBLE PRECISION` | Percentage of winning trades as a decimal |
| `num_transactions` | `INTEGER` | Total number of buy/sell transactions |
| `final_outcome` | `DOUBLE PRECISION` | Final portfolio value minus initial ($1) |
| `final_action` | `VARCHAR(10)` | Final recommended action: `buy`, `sell`, or `hold` |
| `parameters_snapshot` | `JSONB` | Snapshot of strategy parameters at backtest time |
| `created_at` | `TIMESTAMP NOT NULL DEFAULT NOW()` | Record creation timestamp |

**Unique constraint:** `(strategy_id, underlying, start_date, end_date)` — prevents duplicate backtest runs for the same configuration.

**Index:** `(strategy_id, underlying)` for filtered queries.

### FR-03: REST API — Create/Save a Strategy

**Endpoint:** `POST /api/strategies`

**Purpose:** Save a new strategy configuration to the `strategies` table.

**Request Body:**

```json
{
  "name": "My RSI Strategy",
  "strategy_type": "rsi_strategy",
  "underlying": "AAPL",
  "timeframe": "1d",
  "lookback_days": 365,
  "parameters": { "period": 14, "oversold": 30, "overbought": 70 }
}
```

**Validations:**
- `name` is required and must be unique
- `strategy_type` must be one of the known types from FR-01
- `underlying` is required, normalized to uppercase
- `parameters` must be valid JSON

**Response:** `201 Created` with the created strategy object including the generated `id`.

**Authentication:** Bearer token required.

### FR-04: REST API — Generate Signals for a Strategy

**Endpoint:** `POST /api/strategies/{id}/signals`

**Purpose:** Run a saved strategy against OHLCV data and generate buy/sell/hold signals. Signals are persisted in the `signals` table.

**Behavior:**
1. Load the strategy definition from `strategies` table by `id`.
2. Fetch OHLCV snapshots from the `snapshots` table for the strategy's `underlying` asset within the strategy's `lookback_days`.
3. Check if signals already exist for this `strategy_id + underlying + date range`.
   - If yes and `force` is `false`, return existing signals without recomputation.
   - If `force: true` in the request body, delete existing signals and regenerate.
4. Use the cinar/indicator library to instantiate the strategy type with the stored parameters.
5. Compute actions (Buy/Sell/Hold) using `strategy.Compute()` or `strategy.ComputeWithOutcome()`.
6. Insert all signal records into the `signals` table.
7. Return the generated signals.

**Request Body (optional):**

```json
{
  "force": false
}
```

**Response:**

```json
{
  "strategy_id": 1,
  "strategy_name": "My RSI Strategy",
  "underlying": "AAPL",
  "signal_count": 252,
  "signals": [
    { "date": "2026-01-15", "action": "buy", "price": 185.50 },
    { "date": "2026-02-20", "action": "sell", "price": 192.30 },
    ...
  ],
  "cached": true,
  "generated_at": "2026-07-04T00:00:00Z"
}
```

The `cached` field indicates whether signals were freshly computed (`false`) or returned from a previous run (`true`).

**Authentication:** Bearer token required.

### FR-05: REST API — Run Backtest for a Strategy

**Endpoint:** `POST /api/strategies/{id}/backtest`

**Purpose:** Run historical backtesting for a saved strategy and store performance/risk metrics in `backtest_results`.

**Behavior:**
1. Load the strategy definition from `strategies` table by `id`.
2. Fetch OHLCV snapshots for the strategy's underlying asset.
3. Check if a backtest result already exists for this `strategy_id + underlying + start_date + end_date`.
   - If yes and `force` is `false`, return existing result.
   - If `force: true`, delete existing and rerun.
4. Use the cinar/indicator backtest engine (`backtest.NewBacktest` + `backtest.DataReport`) to compute performance metrics.
5. The backtest produces:
   - `total_return` — final portfolio return
   - `max_drawdown` — maximum peak-to-trough decline
   - `sharpe_ratio` — risk-adjusted return (if the library provides it; otherwise compute from outcome stream)
   - `win_rate` — percentage of profitable trades
   - `num_transactions` — count of trades
   - `final_outcome` — final P&L
   - `final_action` — last recommended action
6. Insert or update the `backtest_results` row.
7. Return the performance/risk metrics.

**Request Body (optional):**

```json
{
  "force": false
}
```

**Response:**

```json
{
  "strategy_id": 1,
  "strategy_name": "My RSI Strategy",
  "underlying": "AAPL",
  "start_date": "2025-07-04",
  "end_date": "2026-07-04",
  "total_return": 0.1523,
  "max_drawdown": -0.0831,
  "sharpe_ratio": 1.24,
  "win_rate": 0.58,
  "num_transactions": 24,
  "final_outcome": 0.1523,
  "final_action": "sell",
  "cached": false,
  "generated_at": "2026-07-04T00:00:00Z"
}
```

**Authentication:** Bearer token required.

### FR-08: MCP Tools (Parity with REST Endpoints)

Every REST endpoint must have a corresponding MCP tool registered in the existing MCP handler (`internal/mcp/tools.go`).

| MCP Tool | Maps To | Description |
|----------|---------|-------------|
| `list_strategy_types` | GET /api/strategies/types | List all available strategy types from the library catalog |
| `list_strategies` | GET /api/strategies | List saved strategies with optional filters |
| `create_strategy` | POST /api/strategies | Create/save a new strategy configuration |
| `generate_signals` | POST /api/strategies/{id}/signals | Generate and persist buy/sell/hold signals for a saved strategy |
| `run_backtest` | POST /api/strategies/{id}/backtest | Run backtesting and store performance/risk metrics |
| `query_signals` | GET /api/signals | Query generated signals with filters |
| `query_backtest_results` | GET /api/backtest-results | Query backtest performance/risk evaluations with filters |

Each MCP tool must:
- Use typed input structs with `jsonschema` tags for auto-generated JSON Schema
- Return results as JSON `TextContent`
- Reuse the same engine/package code as the REST handlers (no reimplementation)

## 6. Acceptance Criteria

### AC-01: Strategy Type Listing
- `GET /api/strategies/types` returns 200 with a non-empty list of strategy types grouped by category.
- Each entry includes `type`, `name`, `category`, `description`, and `default_parameters`.
- The MCP tool `list_strategy_types` returns the same data.

### AC-02: Strategy CRUD
- `POST /api/strategies` creates a strategy, returns 201 with the strategy object including `id`.
- `GET /api/strategies` returns all saved strategies.
- Filtering by `strategy_type`, `underlying`, and `name` works correctly.
- Duplicate `name` returns 409 Conflict.
- Unknown `strategy_type` returns 400 Bad Request.
- MCP tools `create_strategy` and `list_strategies` produce identical results.

### AC-03: Signal Generation — Idempotency
- First call to `POST /api/strategies/{id}/signals` generates and stores signals.
- Second call with the same strategy returns `cached: true` and does NOT recompute.
- Calling with `force: true` deletes existing signals and regenerates.
- The `signals` table has no duplicate rows for the same `(strategy_id, underlying, signal_date)`.
- MCP tool `generate_signals` behaves identically.

### AC-04: Signal Generation — Correctness
- Generated signals match the expected behavior of the strategy type (e.g., RSI strategy generates buy when RSI < 30, sell when RSI > 70).
- Signal dates align with trading days from the snapshots data.
- Prices match the closing prices of the snapshots.

### AC-05: Signal Querying
- `GET /api/signals` supports filtering by `strategy_id`, `strategy_type`, `underlying`, `date_from`, `date_to`, `action`.
- Pagination with `limit` and `offset` works.
- MCP tool `query_signals` returns the same data.

### AC-06: Backtesting
- `POST /api/strategies/{id}/backtest` returns performance/risk metrics.
- Metrics include `total_return`, `max_drawdown`, `sharpe_ratio`, `win_rate`, `num_transactions`, `final_outcome`, `final_action`.
- Second call returns `cached: true`.
- `force: true` reruns the backtest.
- The `backtest_results` table captures the data correctly.
- MCP tool `run_backtest` behaves identically.

### AC-07: Backtest Result Querying
- `GET /api/backtest-results` supports filtering by `strategy_id`, `strategy_type`, `underlying`, date range, `min_return`.
- Pagination with `limit` and `offset` works.
- MCP tool `query_backtest_results` returns the same data.

### AC-08: Auth Protection
- All POST endpoints return 401 without a valid Bearer token.
- All GET endpoints work without authentication.

### AC-09: Case Normalization
- Creating a strategy with `underlying: "aapl"` stores it as `"AAPL"`.
- Querying by `underlying: "aapl"` returns results for `"AAPL"`.
- Case-insensitive matching on `underlying` (using `LOWER()` in SQL).

### AC-10: No Regressions
- All existing F-001 through F-004 tests pass.
- All existing MCP tests pass.

## 7. Out of Scope

- **Live trading / order execution.** This feature is about signal generation and paper backtesting only. Connecting to a broker API (Alpaca, Interactive Brokers, etc.) is a future feature.
- **Strategy comparison / ranking.** Comparing multiple strategies side-by-side in a single view is out of scope for the initial release.
- **Portfolio-level backtesting.** Backtesting multiple assets under a single strategy simultaneously is not supported in v1. Each strategy is per-asset.
- **Custom strategy definition via DSL.** Strategies must be chosen from the pre-built library. Users cannot write custom strategy logic through the API.
- **Real-time/scheduled signal generation.** Signal generation is on-demand only. Scheduled/cron-based generation is a future enhancement.
- **Optimization / parameter sweeping.** Automated grid search for optimal strategy parameters is out of scope.
